## Overview

This repository contains the source Markdown files and MkDocs configuration for the findmyclient API documentation site. The site covers everything developers need to integrate with findmyclient: authentication, endpoints, request/response structure, n8n workflow integration, pricing, and resources.

The documentation is:
- Written in **Markdown** inside `docs/`
- Built into a static site by **MkDocs Material**
- Served in production via **Flask**

## Project structure

```
findmyclient-api-docs/
├── docs/                   # Source Markdown files (edit these)
│   ├── index.md            # Landing / home page
│   ├── introduction.md     # What is findmyclient?
│   ├── usage.md            # API usage and examples
│   ├── limitations.md      # Rate limits and constraints
│   ├── resources.md        # External links and tools
│   └── assets/             # Images and static files
│       └── workflow.webp   # n8n workflow diagram
├── mkdocs.yml              # MkDocs configuration
├── site/                   # Generated static site — DO NOT EDIT
└── README.md
```

> **Important:** never manually edit files inside `site/`. That folder is fully generated by `mkdocs build` and will be overwritten on the next build.

## Getting started

### Prerequisites

- Python 3.9+
- pip

### Install dependencies

```bash
pip install mkdocs mkdocs-material
```

---

## Local development

Start the MkDocs development server:

```bash
mkdocs serve
```

Then open your browser at:

```
http://127.0.0.1:8000/
```

The dev server hot-reloads automatically whenever you save a `.md` file — no need to restart.

---

## Building the site

To generate the final static site:

```bash
mkdocs build
```

This creates or updates the `site/` folder with production-ready HTML, CSS, and JS.

Run this before deploying or pushing an update that needs to be served.

---

## How Flask serves the docs

After `mkdocs build`, the generated `site/` folder is served directly by the Flask application in the main [findmyclient-live](https://github.com/Rottie420/findmyclient-live) repo.

URL routing works like this:

| URL | Serves |
|-----|--------|
| `/` | `site/index.html` |
| `/usage/` | `site/usage/index.html` |
| `/resources/` | `site/resources/index.html` |

Flask treats `site/` as a static directory — no additional backend logic is required for the docs.

## Editing documentation

All content lives in the `docs/` folder as Markdown files.

1. Open the relevant `.md` file in `docs/`
2. Make your changes
3. Preview locally with `mkdocs serve`
4. Once happy, run `mkdocs build` to regenerate `site/`
5. Commit both the `docs/` changes and the updated `site/` folder

### Images and assets

- Store images in `docs/assets/`
- Use **WebP format** for faster load times (convert PNGs/JPGs before adding)
- Reference them in Markdown as: `![alt text](assets/your-image.webp)`

---

## Adding new pages

1. Create a new `.md` file inside `docs/`, e.g. `docs/authentication.md`
2. Add it to the `nav` section in `mkdocs.yml`:

```yaml
nav:
  - Home: index.md
  - Introduction: introduction.md
  - Authentication: authentication.md   # ← new page
  - Usage: usage.md
  - Limitations: limitations.md
  - Resources: resources.md
```

3. Run `mkdocs serve` to preview, then `mkdocs build` when ready.

## Deployment

This repo is part of the findmyclient stack. The built `site/` folder is picked up by the Flask app on deploy. After making doc changes:

```bash
mkdocs build
git add docs/ site/ mkdocs.yml
git commit -m "docs: <describe your changes>"
git push
```

The CI/CD pipeline in [findmyclient-live](https://github.com/Rottie420/findmyclient-live) will pick up the updated `site/` on the next deploy.

## Theme

This project uses the [MkDocs Material](https://squidfunk.github.io/mkdocs-material/) theme, configured in `mkdocs.yml`. It includes:

- Dark and light mode toggle
- Sidebar navigation
- Full-text search
- Responsive layout

Theme customisation (colours, logo, favicon) is handled inside `mkdocs.yml`. Do not override theme styles by editing files inside `site/`.

## Contributing

Contributions are welcome — whether that's fixing a typo, clarifying an example, or adding a missing section.

1. Fork the repository
2. Create a branch: `git checkout -b docs/your-change`
3. Edit Markdown files in `docs/`
4. Preview with `mkdocs serve`
5. Build with `mkdocs build`
6. Open a pull request against `main`

Please keep commit messages descriptive and prefixed:
- `docs:` for content changes
- `feat:` for new pages or sections
- `fix:` for corrections
- `chore:` for config, build, or asset updates
- `refactor:` for structural reorganisation
